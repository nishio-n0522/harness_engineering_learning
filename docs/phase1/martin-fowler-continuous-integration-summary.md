## 1. イントロダクション

80年代のイギリスの電子機器メーカーでの体験談から始まる。プログラマーたちが数年コードを書いた後、統合作業に数ヶ月を要し、完了時期が誰にも分からない状態だった。この導入から、複数開発者の作業統合が予測不能な長期プロセスになりがちだという問題が提示される。

現代でも同様の摩擦は残る。メインブランチへの大きな変更に後追いで追随する苦労、プルリクエスト待ち中の再統合、マージ後に走る統合テストでの手戻り、など。

これを避ける鍵は「高価なツール」ではなく、**全員が最低1日1回、管理されたリポジトリに統合する**という単純な実践にある。これを Continuous Integration（またはTrunk-Based Development）と呼ぶ。多くのチームが「CIをやっている」と言いつつ本質を外しているため、用語を正しく理解することが重要。

## 2. CIでの機能開発の実例

架空の「魔法薬システム」で飛行薬の効果持続時間計算を追加する例を使い、1サイクルを具体的に描く。

**ワークフロー：**

1. `git pull` で最新を取得
2. ローカルでビルド＆テストを実行し、変更前に緑であることを確認
3. コード変更と自動テスト追加を並行して進める
4. 再度pullして他者の変更と結合
5. セマンティック競合が発生したら修正
6. push
7. CIサービスが中央環境でビルド
8. 緑の通知を受けて次の作業へ

「works on my machine」問題は稀に起きるが、あり得ないわけではない、という現実的な注釈が入る。

## 3. CIの実践（11項目）

記事の核。11の具体的プラクティスが順に解説される。

### 3.1 バージョン管理されたメインラインに全てを入れる

OSだけ入ったノートPCを持ってきて、リポジトリをクローンすれば全てをビルド・実行できる状態が理想。

**リポジトリから取得できるべきもの：**
- ソースコード、テスト、DBスキーマ、テストデータ
- 設定ファイル、IDE設定、インストールスクリプト
- サードパーティライブラリ、ビルドに必要なツール

ビルド成果物は入れない（コードスメル）。チーム全体で単一の「メインライン」を共有。diffが意味を持つテキスト形式を優先。

### 3.2 ビルドの自動化

人間にGUI操作やコマンド入力をさせるのはミスの温床。ビルド指示は全てリポジトリ内のテキストで管理。`make` 系の**依存ネットワーク型ツール**が向く。DBスキーマ構築まで含めて自動化するのが要点。

### 3.3 セルフテスティングなビルド

手動テストはCIの頻度についていけない。XUnit系フレームワークでテストを書き、「コードに意地悪ないたずらをする小鬼がいたら必ずテストが落ちる」水準を目指す。

**重要な位置づけ：**
- CIの**必須前提**
- CI導入の最大の障壁はしばしばテストスキル不足
- TDDは必須ではないが相性が良い
- リンターやセキュリティスキャナも組み合わせる

### 3.4 全員が毎日メインラインにpushする

統合はコミュニケーションである。頻繁にpushすることで競合を早期発見できる。

**2種類の競合：**
- **テキスト上の競合（マージ競合）：** バージョン管理ツールが検出してくれる
- **セマンティック競合：** 関数名変更や挙動変更による意味の衝突。これこそセルフテストコードが必要な理由

### 補足：Semi-Integration（半統合）はCIではない

ブランチにメインラインの変更を取り込むだけでは「半分」で、自分の変更をメインラインにpushして初めて完全な統合になる。CIは数時間以上ブランチに留めることを許さない。

### 3.5 メインラインへのpushごとにビルドをトリガー

CIサービス（Jenkins / GitHub Actions / Circle CI 等）が**メインラインだけを**監視する。複数ブランチでビルドを走らせるのはCIとは別物。CIサービスの誕生にはThoughtworksのCruise Controlが関わっている、という歴史的注釈あり。

### 3.6 壊れたビルドを即座に直す

ケント・ベックの言葉として「ビルド修正より優先すべき仕事はない」が引用される。

- 全員が手を止める必要はないが、優先度は最高
- 通常は**直前コミットのリバート**が最善の対処
- リスクを下げたければPending Head（ゲート付きコミット）という選択肢もある
- 健全なチームなら頻発しないので可視的な赤信号のままでも良い

### 3.7 ビルドを速く保つ

XPの「**10分ルール**」が基準。遅くする最大要因は外部サービスを伴うテスト。

**解決策：デプロイメントパイプライン**
- 高速な**コミットビルド**で素早くフィードバック
- 本物のDBやE2Eを含む低速な**二次ビルド**を後段で実行
- 二次ビルドで見つかったバグは可能な限りコミットビルドのテストに「昇格」させる

クラウドによる並列化、依存関係（外部ライブラリ）の自動更新とコントラクトテストも言及。

### 3.8 Work-in-Progress（未完成機能）を隠す

未完成でもメインラインに統合するため、本番で実行されない「潜在コード」を安全に置くテクニック群。

| テクニック | 用途 |
| --- | --- |
| **Keystone Interface** | ユーザーに見える最後のインターフェース結線を最後に行う |
| **Feature Flags** | 環境別設定でオンオフ、A/Bテストやカナリアにも使える |
| **Dark Launching** | 本番で実行しつつユーザーには見せない |
| **Branch By Abstraction** | 大規模な基盤変更を内部インターフェース経由で段階移行 |
| **Parallel Change (expand-contract)** | DBフィールド名変更などを可逆ステップに分解 |

### 3.9 本番環境のクローンでテストする

環境差は静かなバグの温床。

**揃えるべき項目：** OS・DB・ライブラリ・IP・ポート・ハードウェア。

コンテナ・仮想化の普及で現実的になった。マルチ環境向け製品は並列テスト。本番が悪環境（モバイルの不安定ネットワーク等）なら、その悪さも再現する。

### 3.10 誰もが状況を見られる

- ビルド状態のダッシュボード、Slack通知、IDE連携
- 成功通知も送ることで「普段の鼓動」が分かる
- 物理ディスプレイの事例：ラバランプ、踊るうさぎなど遊び心のある例
- 1年カレンダーに日ごとの緑／赤シールを貼って改善を可視化した事例

### 3.11 デプロイの自動化

複数環境への配布を自動化。コンテナとInfrastructure as Codeで環境自体も自動構築。同じ仕組みで本番にも流せる。

**派生して得られるもの：**
- 自動ロールバック
- Blue/Green Deployment
- Canary Release
- モバイルや組み込みデバイスへの自動配布
- 実行中のバージョン情報の可視化（about画面、ログ、APIエンドポイント）

## 4. 統合スタイルの3類型

| スタイル | 特徴 | 統合頻度 |
| --- | --- | --- |
| **Pre-Release Integration** | ウォーターフォール的。リリース直前に一括統合フェーズ | 月〜年単位 |
| **Feature Branches** | 機能ごとに分岐、完成時にメインラインへ統合。現在最も普及 | 機能単位 |
| **Continuous Integration** | 日次で全員が双方向に統合 | 1日複数回 |

重要な整理として、FBはメインラインの変更を取り込んではいるものの、自分の未完成作業を他人に見せていないため**半統合**にとどまる、という論点が再提示される。

## 5. CIの5つの恩恵

### 5.1 デリバリー遅延リスクの低減

統合時間は予測困難で、統合期間が長いほど時間の**非線形的増加**と不確実性が増す。プレリリース統合はリリース直前の圧力下で起きるため最悪。CIでは統合が数分単位で済むため、実質的に遅延リスクがなくなる。

### 5.2 統合に費やす時間そのものの減少

コード量が2倍なら統合時間は4倍以上になるような非線形関係。FBではこの負担が個人に集中する。

**個人にかかる負荷の例：**
- リベース作業
- PRレビュー待ち中の再整合
- 2週間前の機能のデバッグ

CIでは統合が「非イベント」になる。**よくある罠：** 痛い統合を経験して「統合頻度を下げよう」という逆方向に学習してしまう。

### 5.3 バグの減少

主因はCI自体より、その前提である**セルフテストコード**。

- 頻繁な統合は小さな差分なのでセマンティック競合も発見しやすい
- Diff Debuggingが効きやすい
- バグは累積するほど相互作用と心理的諦めで除去が難しくなる

**DORA研究（Nicole Forsgren）の裏付け：**
- デプロイ頻度を上げるほど失敗率が下がる（反直感的）
- アクティブブランチ3本以下・日次マージ・コードフリーズなしのチームが高パフォーマンス

### 5.4 持続可能な生産性のためのリファクタリングを可能にする

コードベースは時間とともに劣化する。2週間のリファクタリングブランチは統合地獄になって誰もやりたがらなくなり、結果として技術的負債が蓄積する。

CIは「リファクタ担当者と他の全員が常に同期している」状態を作ることでこの罠を解く。

**最重要の反直感的原則：コード品質への投資が配信速度を生む。**

### 5.5 本番リリースがビジネス判断になる

**Release-Ready Mainline（いつでもリリース可能なメインライン）** を維持することで、ステークホルダーが「今すぐ出したい」と言ったときに数分で実現可能。開発者と顧客の距離を縮める効果が最も大きい。

## 6. CIを使うべきでない場合

前述の恩恵はほぼトレードオフなしで得られるが、CIが合わないのは**「コミットしていない、またはスキル不足の」チーム**。

**コミットの欠如の例：**
古典的なオープンソース。メンテナが週数時間しか触れず、貢献者を知らず、標準を共有できないためFB＋PRが自然。

**スキル不足の例：**
- セルフテストコードがない
- 自動化不足
- 規律不足（赤のままpushする）

**重要な注意：** ロックスター開発者が必要なわけではない。むしろロックスター的な人は規律不足で障害になりがち。技術的スキルよりも**良き師と習慣化**が鍵。

## 7. CIの導入

導入経路はチームの現状に依存するため一律の処方はない。

**導入時のポイント：**
- **なぜ導入するか**を明確にする（特に「リファクタリングを可能にする」という最重要かつ見えにくい価値）
- セルフテストコードは統合頻度を上げる前から価値を生む
- 現実的な目標として「統合頻度を倍にする」（10日なら5日に）
- 新規プロジェクトなら最初から10分ルールを監視
- **経験者の支援を得る**ことが近道

## 8. よくある質問（8項目）

### 8.1 CIの起源

1990年代のケント・ベック（XP）が定義し、実践として確立した。Microsoftのデイリービルドはテストと修正規律がないため別物。Grady Boochは著書で一言使ったのみで、実践として定義していない。

### 8.2 CIとTrunk-Based Developmentの違い

実質同義。セマンティック拡散を避けるために造られた別名だが、ファウラーは元の用語を使う（ケント・ベックの貢献を消さないため）。

### 8.3 フィーチャーブランチでCIサービスを動かすのはCI？

**いいえ、それは半統合。** ツール名に引きずられた誤解で、本来は「Continuous Build Service」と呼ぶべき。

### 8.4 CIとFBを同時にできるか？

基本的に相互排他。例外は全機能が1日未満で終わるチームだが、それは実質CI。個人的にローカルブランチで作業してから統合するのは自由で、鍵は「継続的に統合しているか」。

### 8.5 CIとContinuous Deliveryの違い

- **CI：** 開発チームの統合サイクル
- **CD：** 統合後の本番までのパス

解釈には幅があり、CDがCIを包含するとする人もいれば、同義扱いの人もいる。

### 8.6 Continuous Deploymentとの関係

階層関係：

```
CI（日次統合）
  → Continuous Delivery（いつでもリリース可能な状態維持）
    → Continuous Deployment（パイプライン通過で自動本番デプロイ）
```

Continuous DeliveryはContinuous Deploymentの前提。

### 8.7 プルリクエストとコードレビュー

PRはGitHubが広めたもので、OSSの文脈から来ている。事前レビューはCIの摩擦源になる。

**対策：**
- ペアプログラミングでリアルタイムレビュー
- Ship/Show/Askプロセスで必要な時だけブロック
- **Refinement Code Review**（事後レビュー）

少人数のフルタイムチームではOSS由来の慣行を再考すべき。

### 8.8 データベースの扱い

従来は大きな課題だったが、Pramod Sadalageらの **Evolutionary Database Design** で解決。

- 小さなマイグレーションスクリプトの連鎖としてスキーマとデータを管理
- データアクセスコードと同期してバージョン管理
- マイグレーションはテストデータ・本番DBの両方に適用可能

## 9. 最終的な考察

ソフトウェア開発は主に既存コードの改変であり、コードベースの健全性が変更コストを決める。

**三位一体の関係：**
- 健全性を保つには**リファクタリング**が必要
- リファクタリングを支えるのは**セルフテストコード**
- そしてこれらを機能させるのが**Continuous Integration**

XPから生まれたこれらの実践は、変化する要求と技術機会に応じて製品を継続進化させる基盤である。

## 学習上のポイント（自分用メモ欄）

- CIの定義で最も誤解されやすいのは「フィーチャーブランチにCIサービスを動かすのもCI」という勘違い。**メインラインへの日次push**が核であることを忘れない
- セルフテストコードが前提条件であり、テストスキル不足がCI導入の最大障壁
- 「品質への投資が速度を生む」という反直感的原則が、記事全体を貫く最重要メッセージ
- ハーネスエンジニアリング学習プランのフェーズ1（lefthook）とフェーズ4（CI/CD）の背景思想として、この記事の論点が効いてくる
