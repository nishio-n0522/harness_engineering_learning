# シェルスクリプトの基本

シェルスクリプトを「他言語の延長」で読もうとすると必ずどこかで詰まります。理由はシェルが**値ではなく終了コードで会話する言語**だからです。本ドキュメントはこの世界観を軸に、初学者がつまずきがちなポイント（クォート、変数、終了コード、`true`/`false`、制御構造）を体系的にまとめたものです。

対象シェルは POSIX sh / bash を想定しています。

## 1. シェルの世界観

他の言語と決定的に違う点を最初に押さえます。

### 値ではなく終了コードで会話する

JavaScriptやPythonの `if` は**値の真偽**で分岐します。

```js
if (true) { ... }
```

シェルの `if` は**コマンドの終了コード**で分岐します。

```sh
if コマンド; then
  ...
fi
```

`if` の後ろに書くのは「真偽値」ではなく「コマンド」です。コマンドが終了コード0を返したら `then`、非ゼロなら `else`。

### すべては文字列とコマンド

シェルが扱うデータ型は実質的に**文字列だけ**です。数値も日付も真偽値も、すべて文字列として保持され、必要に応じてコマンドが解釈します。

そして「処理」はすべて**コマンドの実行**として表現されます。`if` `while` `[` `true` `false` も、多くがコマンドです。

この2点を頭に入れておくと、後の話が一貫して見えてきます。

## 2. コマンド実行の基本

### 最も単純な形

```sh
コマンド名 引数1 引数2 ...
```

スペースが区切りです。

```sh
echo hello world
ls -la /tmp
mkdir -p path/to/dir
```

### shebang（シェバン）

スクリプトの1行目に書く `#!` はインタプリタ指定です。

```sh
#!/bin/sh
#!/bin/bash
#!/usr/bin/env bash
```

`#!/usr/bin/env bash` 形式は環境変数 `PATH` から `bash` を探すので、移植性が高くなります。

### 実行権限

スクリプトは実行権限がないと動きません。

```sh
chmod +x myscript.sh
./myscript.sh
```

権限がない場合は `sh myscript.sh` のようにインタプリタ経由で実行することもできます。

### コメント

`#` 以降は行末までコメント。

```sh
# これはコメント
echo hello   # ここもコメント
```

## 3. 終了コード（exit status）

コマンドは実行後に必ず**0〜255の整数**を返します。

| 終了コード | 意味                                                     |
| ---------- | -------------------------------------------------------- |
| `0`        | 成功                                                     |
| `0以外`    | 失敗、または何らかの情報（コマンドによって意味が異なる） |

直前のコマンドの終了コードは特殊変数 `$?` で取れます。

```sh
ls /etc > /dev/null
echo $?   # → 0

ls /存在しないパス 2> /dev/null
echo $?   # → 2
```

### 自分のスクリプトから終了コードを返す

```sh
exit 0    # 成功で終了
exit 1    # 失敗で終了
```

`exit` を省略するとスクリプトは「最後に実行されたコマンドの終了コード」で終わります。

### コマンドによって意味が違うことに注意

- `grep`：マッチあり=0、マッチなし=1、エラー=2
- `diff`：差分なし=0、差分あり=1、エラー=2
- `test`（`[`）：真=0、偽=1

「成功/失敗」と単純に二値で考えず、コマンドのmanを読む習慣をつけると安全です。

## 4. `true` / `false` / `:` の正体

### キーワードではなくコマンド

これが他言語経験者の最大の混乱ポイントです。

```sh
which true
# → /usr/bin/true
```

`true` と `false` は**実在するコマンド**で、それぞれ「何もせず終了コード0/1を返すだけ」のミニマムなプログラムです。ブール値リテラルではありません。

```sh
true
echo $?   # → 0

false
echo $?   # → 1
```

### 単独で書いた場合

```sh
echo "before"
false
echo "after"
```

実行結果：

```
before
after
```

`false` を書いても次の行は実行されます。非ゼロ終了コードを返すだけで、シェルは勝手には止まりません（`set -e` を付けない限り）。

### `:` というヌルコマンド

`:`（コロン）は `true` とほぼ同じで、「何もせず0を返す」コマンドです。歴史的には `:` のほうが古い。空ブロックのプレースホルダとしてよく使います。

```sh
if [ "$DEBUG" = "yes" ]; then
  :   # TODO: あとで実装
fi
```

### 使いどころ

```sh
# 無限ループ
while true; do
  echo "ずっと回る"
  sleep 1
done

# エラー握り潰し（true 版）
command_that_might_fail || true

# エラー握り潰し（: 版、動作は同じ）
command_that_might_fail || :

# テスト時のプレースホルダ
if some_condition; then
  :
fi
```

## 5. クォートの役割

クォートの本質は「**シェルの特殊解釈を抑制する**」ことであって、「文字列であることを示す」ことではありません。

### 3種類のクォート

| 記法                | 変数展開 | コマンド置換 | ワイルドカード | バックスラッシュ |
| ------------------- | -------- | ------------ | -------------- | ---------------- |
| クォートなし        | 有効     | 有効         | 有効           | 有効             |
| `"..."`（ダブル）   | 有効     | 有効         | 無効           | 有効             |
| `'...'`（シングル） | 無効     | 無効         | 無効           | 無効             |

### 例で見る

```sh
name=Alice

echo $name        # → Alice
echo "$name"      # → Alice
echo '$name'      # → $name（展開されない）

echo *.txt        # → カレントの .txt ファイル一覧
echo "*.txt"      # → *.txt（そのまま）
```

### スペース処理

```sh
echo hello world      # 引数2つ
echo "hello world"    # 引数1つ
```

ファイル名にスペースが含まれるケースで一番事故ります。

```sh
file="my file.txt"
rm $file       # → "my" と "file.txt" の2ファイルを消そうとする（事故）
rm "$file"     # → "my file.txt" を消す（正解）
```

**変数を展開するときは原則ダブルクォートで囲む**、と覚えておくと安全です。

## 6. 変数の取り扱い

### 代入

```sh
name=Alice
count=10
path=/usr/local/bin
```

**`=` の前後にスペースを入れてはいけません**。これは構文エラーになります。

```sh
name = Alice    # エラー：「name」というコマンドを探しに行く
```

スペース禁止は、シェルが「コマンド名 引数」のパターンで解釈する以上避けられない仕様です。

### 参照

`$` を付けて展開します。

```sh
name=Alice
echo $name        # → Alice
echo "$name"      # → Alice（推奨：クォート付き）
echo "${name}"    # → Alice（明示的境界）
```

### `${変数名}` の波括弧の意味

変数名の境界を明示します。

```sh
name=Alice
echo "$nameSan"     # → 何も出ない（nameSan という変数を探す）
echo "${name}San"   # → AliceSan
```

### よくある特殊変数

| 変数            | 意味                              |
| --------------- | --------------------------------- |
| `$0`            | スクリプト名                      |
| `$1`, `$2`, ... | 位置引数                          |
| `$#`            | 引数の個数                        |
| `$@`            | 全引数（個別に展開）              |
| `$*`            | 全引数（1つの文字列として）       |
| `$?`            | 直前のコマンドの終了コード        |
| `$$`            | 現在のシェルのPID                 |
| `$!`            | 直前にバックグラウンド実行したPID |

### 環境変数

`export` で子プロセスに引き継がれます。

```sh
MY_VAR=hello             # シェル内ローカル
export MY_VAR=hello      # 子プロセスにも引き継がれる
export PATH="$PATH:/opt/bin"
```

### デフォルト値の指定

```sh
echo "${VAR:-default}"     # VAR が未設定または空なら default を使う
echo "${VAR:=default}"     # 同じ。さらに VAR に default を代入する
echo "${VAR:?エラーメッセージ}"  # 未設定なら停止
echo "${VAR:+代替値}"        # VAR が設定されているなら代替値を使う
```

実務でよく使うのは `:-` です。

```sh
LOG_LEVEL="${LOG_LEVEL:-info}"
```

## 7. 変数展開とコマンドの境界

ここが「`true` は文字列なのかコマンドなのか」問題の核心です。

### シェルは「位置」で意味を変える

同じ語でも、置かれた位置で解釈が変わります。

| 書き方   | 文脈                      | 解釈                             |
| -------- | ------------------------- | -------------------------------- |
| `true`   | コマンド位置              | `true` コマンドを実行            |
| `true`   | 引数位置（`echo true`）   | 文字列「true」を引数として渡す   |
| `true`   | 代入の右辺（`flag=true`） | 文字列「true」を変数に代入       |
| `"true"` | どこでも                  | 文字列「true」（クォートで保護） |

### 変数の中身が「コマンドとして再解釈される」

```sh
flag=true

if $flag; then
  echo "通る"
fi
```

ステップ：

1. `flag=true` で文字列 `"true"` を変数に代入
2. `$flag` が変数展開されて、行が `if true; then` になる
3. シェルがこの結果を再解釈し、`true` コマンドを実行

つまり**「文字列が変数に入っている」と「その文字列がコマンドとして実行される」は別の話**です。後者は変数展開後にシェルがその位置をどう解釈するかで決まります。

### 強力さと危険性

```sh
flag=ls
$flag      # → ls が実行される

cmd="echo hello"
$cmd       # → hello
```

文字列がコマンドラインとして再評価される、これがシェルの動的な性質です。便利ですが、ユーザー入力をそのまま展開するとコマンドインジェクションになるので注意。

### 安全な比較

ブール的な値を扱うときは文字列比較のほうが推奨されます。

```sh
flag=true

# 動くが意図が読みにくい
if $flag; then ... fi

# こちらが安全で意図が明確
if [ "$flag" = "true" ]; then ... fi
```

## 8. 制御構造

### if / elif / else

```sh
if コマンド; then
  ...
elif コマンド; then
  ...
else
  ...
fi
```

`if` の後ろは「コマンド」であって、真偽値ではありません。

### test コマンド `[ ... ]`

条件比較に使う `[` は実は**コマンド**です。`/usr/bin/[` が実在します。だから前後にスペースが必要です。

```sh
[ "$x" = "hello" ]       # OK
["$x" = "hello"]         # エラー：「[hello」というコマンドを探す
```

主な比較演算子：

| 演算子                              | 意味             |
| ----------------------------------- | ---------------- |
| `-eq` `-ne` `-lt` `-le` `-gt` `-ge` | 数値比較         |
| `=` `!=`                            | 文字列比較       |
| `-z`                                | 文字列が空       |
| `-n`                                | 文字列が非空     |
| `-e`                                | ファイルが存在   |
| `-f`                                | 通常ファイル     |
| `-d`                                | ディレクトリ     |
| `-r` `-w` `-x`                      | 読み書き実行可能 |

例：

```sh
if [ -f "$file" ]; then
  echo "ファイル存在"
fi

if [ "$count" -gt 10 ]; then
  echo "10より大きい"
fi

if [ -z "$VAR" ]; then
  echo "VAR は空"
fi
```

### bash 拡張の `[[ ... ]]`

bashでは `[[ ... ]]` という拡張版があり、より安全です（クォート不要、パターンマッチ可能、`&&` `||` が使える）。

```sh
if [[ "$str" == *.txt ]]; then
  echo "txt ファイル"
fi
```

POSIX sh で動かす必要がない場合はこちらを推奨。

### for ループ

```sh
for i in 1 2 3; do
  echo "$i"
done

for file in *.txt; do
  echo "処理中: $file"
done

# C言語風（bash拡張）
for ((i=0; i<10; i++)); do
  echo "$i"
done
```

### while / until

```sh
count=0
while [ "$count" -lt 5 ]; do
  echo "$count"
  count=$((count + 1))
done

# until は条件が偽の間ループ
until [ "$count" -ge 10 ]; do
  count=$((count + 1))
done
```

### case

```sh
case "$1" in
  start)
    echo "起動"
    ;;
  stop)
    echo "停止"
    ;;
  *)
    echo "不明なコマンド"
    ;;
esac
```

`*)` がデフォルトケース。

### 関数

```sh
greet() {
  echo "Hello, $1!"
  return 0
}

greet "Alice"      # → Hello, Alice!
```

関数の戻り値は**終了コード**（0〜255の整数）で、文字列を返したい場合は `echo` で出力して呼び出し側で `$()` で受け取ります。

```sh
get_name() {
  echo "Alice"
}

name=$(get_name)
echo "$name"
```

## 9. パイプとリダイレクト

### パイプ `|`

前のコマンドの標準出力を、次のコマンドの標準入力に渡します。

```sh
ls | grep ".txt"
cat file.log | sort | uniq -c | sort -rn | head
```

### リダイレクト

| 記法          | 意味                                   |
| ------------- | -------------------------------------- |
| `> file`      | 標準出力をファイルに書き込み（上書き） |
| `>> file`     | 標準出力をファイルに追記               |
| `< file`      | ファイルを標準入力にする               |
| `2> file`     | 標準エラー出力をファイルに             |
| `2>&1`        | 標準エラーを標準出力に統合             |
| `> file 2>&1` | 標準出力もエラーも同じファイルに       |
| `&> file`     | 同じ意味（bash拡張）                   |
| `> /dev/null` | 出力を捨てる                           |

例：

```sh
command > out.log 2>&1       # 全出力をログに
command > /dev/null 2>&1     # 全部捨てる（静かに実行）
command 2> /dev/null         # エラーだけ捨てる
```

### ヒアドキュメント

複数行の文字列を渡したいとき。

```sh
cat <<EOF > config.txt
server: localhost
port: 8080
EOF
```

`<<-EOF` にすると先頭のタブが除去されるのでインデントしたいときに便利。

## 10. `&&` と `||`：論理演算子

これは「短絡評価のチェーン」で、シェルでは頻出のイディオムです。

### `&&`：成功したら次を実行

```sh
mkdir -p backup && cp file.txt backup/
```

`mkdir` が成功したときだけ `cp` が実行されます。

### `||`：失敗したら次を実行

```sh
command || echo "失敗しました"
```

### チェーン全体の終了コード

```sh
command || true
```

これは前回の解説の核心です。

- `command` が成功 → `true` は評価されない、全体の終了コードは0
- `command` が失敗 → `true` が実行される、`true` は0を返すので全体も0

つまり「失敗してもスクリプトを止めない」ためのイディオム。`set -e` を付けたスクリプトで、失敗が許容できる箇所に挟みます。

### `if` の代わりに使うパターン

シンプルな条件処理は `&&` `||` で簡潔に書けます。

```sh
[ -f "$file" ] && echo "ファイルあり"
[ -f "$file" ] || echo "ファイルなし"
```

## 11. コマンド置換 `$(...)`

コマンドの**標準出力を文字列として取り込む**仕組みです。

```sh
today=$(date +%Y-%m-%d)
echo "今日は $today です"

count=$(ls | wc -l)
echo "ファイル数: $count"
```

### バッククォート記法（古い書き方）

```sh
today=`date +%Y-%m-%d`
```

POSIXでは現役ですが、ネスト時に読みにくく、エスケープが面倒なので**`$()` を推奨**します。

```sh
# ネストの読みやすさが段違い
result=$(echo $(date) - $(whoami))
```

### よくあるパターン

```sh
# Gitで現在のブランチ名
branch=$(git branch --show-current)

# ステージされたファイル一覧
files=$(git diff --cached --name-only)

# プロセスのPID
pid=$(pgrep -f myapp)
```

## 12. 安全なスクリプトのための定石

本番運用するスクリプトは、以下を冒頭に書いておくのが定石です。

```sh
#!/bin/sh
set -euo pipefail
```

### `set -e`（errexit）

非ゼロ終了コードを検出したら即座にスクリプトを終了。エラーを静かに通り過ぎることを防ぐ。

```sh
set -e
false                # ここで死ぬ
echo "ここには来ない"
```

### `set -u`（nounset）

未定義変数を参照したら即エラー。タイポ事故を防ぐ。

```sh
set -u
echo "$UNDEFINED"    # エラーになる
```

### `set -o pipefail`

パイプの**どこか一つでも**失敗したら全体を失敗扱いにする。

```sh
# pipefail なし
false | true
echo $?              # → 0（最後の true の終了コード）

# pipefail あり
set -o pipefail
false | true
echo $?              # → 1
```

### 注意：厳格モードと grep の組み合わせ

`set -e` や `pipefail` を付けると、**grep が「マッチなし」で終了コード1を返した瞬間にスクリプトが死にます**。これは次の章で詳しく扱います。

### `IFS` の固定（応用）

ファイル名にスペースが入ったときの単語分割対策で、より厳密にやるなら：

```sh
IFS=$'\n\t'
```

これでスペース分割を避け、改行とタブだけで区切るようになります。Google Shell Style Guide なども推奨。

## 13. grepの罠と回避策（実例）

### grepの3つの終了コード

| 終了コード | 意味                                         |
| ---------- | -------------------------------------------- |
| `0`        | 1行以上マッチした                            |
| `1`        | **マッチする行がなかった**（エラーではない） |
| `2`        | 実際のエラー（ファイルが読めない等）         |

「マッチなし = エラー扱い（非ゼロ）」が罠の正体です。

### 何が起きるか

```sh
#!/bin/sh
set -e

files=$(git diff --cached --name-only --diff-filter=ACM | grep '\.js$')
echo "staged js: $files"
```

　　　　　　2. パイプの `grep '\.js$'` がマッチなしで終了コード1を返す3. パイプ全体の終了コードが1（または `pipefail` で1）4. `set -e` が「非ゼロだ、終了！」と判断
5　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　. `echo` に到達せずにスクリプトが死ぬ

　
最悪なのは、**開発中に気づかず本番で初めて顕在化する**こと。

### 回避策

**案1：`|| true` で握る（最頻出）**

```sh
files=$(git diff --cached --name-only --diff-filter=ACM | grep '\.js$' || true)
```

意図が明確で短い。grepの本物のエラー（コード2）も握ってしまう点は割り切り。

**案2：`|| :` でも同じ**

```sh
files=$(git diff --cached --name-only --diff-filter=ACM | grep '\.js$' || :)
```

`:` は `true` のヌルコマンド版。タイプ数が少ない。

**案3：ifで判定**

```sh
files=$(git diff --cached --name-only --diff-filter=ACM)
if echo "$files" | grep -q '\.js$'; then
  js_files=$(echo "$files" | grep '\.js$')
fi
```

冗長だが意図が明確。`grep -q` は出力を抑制するオプション。

**案4：一時的に厳格モードを外す**

```sh
set +e
files=$(... | grep '\.js$')
set -e
```

範囲が見える化される。

### 教訓

シェルでは「コマンドの仕様」と「`set -e` の挙動」と「パイプの終了コード」が絡み合うため、**それぞれの動きを個別に理解しておかないと予測できないバグ**が生まれます。grep問題はその典型例です。

## 14. チートシート

### 起動時の定石

```sh
#!/bin/sh
set -euo pipefail
IFS=$'\n\t'
```

### 変数

```sh
name=Alice                  # 代入（=の前後にスペース禁止）
echo "$name"                # 参照（クォート推奨）
echo "${name}San"           # 境界明示
echo "${VAR:-default}"      # デフォルト値
export PATH="$PATH:/opt"    # 環境変数
```

### 終了コード

```sh
echo $?                     # 直前の終了コード
exit 0                      # 成功で終了
exit 1                      # 失敗で終了
true                        # 0を返すコマンド
false                       # 1を返すコマンド
:                           # ヌルコマンド（true相当）
```

### 制御構造

```sh
if [ "$x" = "hello" ]; then ...; fi
for i in *.txt; do ...; done
while [ "$n" -lt 10 ]; do ...; done
case "$x" in foo) ...;; *) ...;; esac
```

### 比較

```sh
[ "$x" = "y" ]              # 文字列等価
[ "$x" != "y" ]             # 文字列非等価
[ "$n" -eq 10 ]             # 数値等価
[ "$n" -gt 10 ]             # 数値大
[ -z "$x" ]                 # 空文字
[ -n "$x" ]                 # 非空
[ -f "$file" ]              # ファイル存在

```

### パイプとリダイレクト

```sh
cmd1 | cmd2                 # パイプ

cmd 2> err.log              # エラー出力
cmd > out 2>&1              # 全部1ファイル

```

### 論理演算子

```sh
cmd1 && cmd2                # cmd1成功時にcmd2
```

## 学習のヒント

シェルは「動くけど、なぜ動くか説明できない」状態が長く続きやすい言語です。理解を深めるには：

- **`echo` で展開結果を確認する習慣**：変数やコマンド置換を疑った時、`echo` を挟んで実際に何になっているか見る
- **`set -x` でトレース**：スクリプト実行時にすべてのコマンドを表示。`bash -x script.sh` でも可
- **`shellcheck` を使う**：静的解析ツール。クォート抜けや不安全な書き方を指摘してくれる。CIに組み込むのも推奨
- **POSIX sh と bash の違いを意識する**：`[[ ]]`、配列、`${var//pattern/replace}` などはbash拡張。`#!/bin/sh` で書くなら使えない

シェルスクリプトは「**位置と文脈で意味が変わる**」「**終了コードで会話する**」という2つの軸を理解できれば、あとは慣れの問題です。手を動かしながら、想定外の挙動に遭遇したら都度メモを残していくのが一番の上達法だと思います。
